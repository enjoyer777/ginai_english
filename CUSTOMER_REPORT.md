## 1. Что сделано

Запущен и развёрнут AI-ассистент в Telegram — `@ginai_english_bot`. Он работает 24/7, отвечает на типовые вопросы клиентов о курсах, ценах, расписании и преподавателях, квалифицирует горячих лидов, собирает контактные данные и создаёт сделки в Битрикс24 с кратким резюме разговора.

Закрыты все функциональные требования из брифа: ответы по базе знаний с поддержкой разных формулировок одного вопроса, не-выдумывание данных вне базы, квалификация по 4 слотам (цель/уровень/срок/готовность), передача горячих лидов в CRM с уведомлением менеджеров в групповом чате, корректная работа в нерабочее время (без обещаний мгновенного ответа), guardrails против jailbreak-попыток и off-topic.

Доставлено:
- **Telegram-бот** на VPS с авто-рестартом и логированием
- **База знаний в xlsx** на Яндекс.Диске — пять листов (Курсы, Расписание, Преподаватели, FAQ, Настройки), плюс опциональный лист **Праздники** для дат-исключений (праздники, переносы, сокращённые дни)
- **Интеграция с Битрикс24** через входящий вебхук с кастомным полем для дедупликации по Telegram user_id
- **Уведомления в групповой чат менеджеров** в рабочее время, с дедупом раз в час
- **CI/CD** через GitHub Actions — автодеплой на VPS при пуше в `main`
- **Документация** — README, инструкция по эксплуатации (OPERATIONS.md), 16-секционный чек-лист ручного тестирования с **85 пунктами** и **89 юнит-тестами**
- **CLI-утилиты** для повседневной эксплуатации (сброс памяти, разовый дамп пользователей)

## 2. Ключевые технические решения и инженерные «фишки»

### 2.1 Память между сессиями

Бот помнит каждого клиента **постоянно по Telegram user_id**. Если клиент задал вопрос вчера и вернулся сегодня — бот не задаёт повторно те же вопросы (цель обучения, уровень), помнит ранее назначенный телефон и имя, продолжает диалог с того места, на котором остановились.

История диалога хранится в локальной SQLite-БД на VPS. В контекст для LLM подаётся скользящее окно из 20 последних пар «вопрос-ответ» — этого достаточно, чтобы поддержать осмысленный разговор, но не настолько много, чтобы накручивать стоимость токенов.

**Зачем это важно:** клиенты часто думают между сессиями, возвращаются «доспросить» — и для них критично, чтобы бот не начинал диалог с нуля.

### 2.2 Свежесть базы знаний без перезапусков

Заказчик меняет цены, расписание, описания курсов и часы работы прямо в xlsx на Яндекс.Диске — без участия разработчика. Бот подхватывает изменения **в течение 10 минут** (TTL кэша) автоматически, без перезапуска контейнера. В логах виден факт каждого обновления — можно убедиться, что правка дошла.

Парсер xlsx сделан на **Rust-библиотеке calamine** — она терпимее, чем стандартный openpyxl, к файлам, экспортированным Яндекс.Документами / Google Sheets / другими онлайн-редакторами (которые часто отдают xlsx с битым `styles.xml`). Если calamine по какой-то причине не справится, есть fallback на openpyxl с автоматической починкой битого xlsx «на лету».

### 2.3 Сбор телефона: жёсткий гарантированный шаг

В первой версии бот мог посчитать, что Telegram-username — достаточный контакт, и не запрашивать телефон. Менеджерам это создавало проблему: позвонить-то нельзя. Мы это поправили: **бот не передаёт лида в CRM, пока не получит явный номер телефона**. Логика реализована в коде на уровне tool-гарда, а не на «уговоры» LLM — даже если модель «забудет», запрос телефона произойдёт.

Бот понимает номер в любом формате: `+7 999 ...`, `8 (999) ...`, `89991234567`, международные номера. Нормализует в E.164 (`+79991234567`) перед сохранением.

### 2.4 Календарный контекст в системном промпте

GPT-4o-mini известна слабым рассуждением о датах («понедельник — это какое число?»). Мы это решили **подкладывая готовый календарный блок** в системный промпт при каждом запросе. LLM получает уже вычисленные:
- сегодняшнее число и день недели
- текущий статус (рабочее/нерабочее, точные часы)
- завтрашняя дата с днём недели
- расписание на неделю по дням
- ближайшие особые даты (праздники, сокращённые дни)
- ближайшее доступное рабочее окно с конкретной датой и часами

LLM не считает даты сама — она цитирует готовые. Это снимает ~80% типичных ошибок типа «менеджер свяжется в понедельник 12 мая» (когда понедельник — 11 мая).

### 2.5 Защита от устаревших цен в истории

Тонкий момент: если клиент спрашивал «сколько стоит» вчера и бот ответил «36000», а сегодня цена в xlsx сменилась на «33333» — бот может «по инерции» процитировать старый ответ из истории. Мы это явно запрещаем в промпте: **при каждом вопросе про цены, расписание, даты, преподавателей — обязательный повторный вызов tool, не доверять собственным прошлым сообщениям**. Из истории берутся только профиль клиента (цель, уровень) — не числа.

### 2.6 Двух-серверная архитектура для обхода блокировок и защиты ПДн

OpenAI блокирует запросы с российских IP-адресов на уровне страны (HTTP 403). Часть IP-адресов Telegram у некоторых российских провайдеров тоже периодически фильтруются. Мы решили это **прокси-сайдкаром Xray** на не-российском VPS заказчика:

```
[Клиент в Telegram]
        ▼
[Telegram API]
        ▼
[RU VPS]  ── Битрикс24 (РФ)       — прямая связь, ПДн остаются в РФ
   │
   │      Яндекс.Диск (РФ)        — прямая связь
   │
   │      OpenAI (США)
   ▼
[Xray-сайдкар на не-RU VPS]       — только маскированные данные (см. 2.7)
        ▼
[api.openai.com]
```

Архитектура построена так, что **сам по себе non-RU прокси-сервер используется только как сетевой туннель** — он не хранит данные, не процессит их, прокачивает обычный TLS-трафик. Это позволяет обойти страновые блокировки без передачи реальных ПДн за границу (см. п. 2.7).

### 2.7 Маскирование ПДн перед отправкой в OpenAI

**Это, пожалуй, самое важное инженерное решение** с точки зрения соответствия 152-ФЗ.

Сервера OpenAI находятся в США — это юрисдикция, не признаваемая Роскомнадзором «адекватной» для трансграничной передачи ПДн. Без специальных мер любая отправка имени/телефона/email в OpenAI считается нарушением.

Мы реализовали **pre-extraction и маскирование**:

1. **На входе** — каждое сообщение клиента сначала проходит через регулярные выражения, которые ищут телефонные паттерны и email-адреса.
2. Найденный телефон **извлекается и сохраняется на стороне бота** в локальной SQLite БД (РФ). В тексте сообщения номер заменяется на токен `<phone>`.
3. Имена клиентов (которые они называют в диалоге) сохраняются по той же схеме — в БД остаётся реальное имя для отображения в Битриксе, а в историю для LLM подставляется `<имя>`.
4. **Только эта маскированная версия** отправляется в OpenAI. Реальные цифры и слова никогда не покидают РФ.
5. LLM работает с «обезличенным» диалогом: видит, что клиент дал контакт (через `<phone>`), но не видит сам номер. Этого достаточно для квалификации и принятия решения о передаче в CRM.

```
Клиент пишет: "Меня зовут Иван, мой номер +79991234567"
             │
             ▼ pre-extractor (RU VPS, regex)
             │
             ├── сохраняет "+79991234567" в SQLite (РФ)
             ├── сохраняет "Иван" в SQLite (РФ)
             │
             ▼ маскирование
             │
"Меня зовут <имя>, мой номер <phone>"  ← это уходит в OpenAI
```

В Битриксе24 (РФ) хранятся **полные данные**, и клиент в Telegram видит свои данные нетронутыми — маскирование происходит только на пути в OpenAI.

Двойная защита: маскирование на входе **и** повторное маскирование при формировании контекста из истории — даже если в БД когда-то что-то проскочило не маскированным, оно не уйдёт в OpenAI.

## 3. Что осталось на стороне заказчика для полного соответствия 152-ФЗ

Технические меры (п. 2.7) закрывают **значительную, но не всю** ответственность по 152-ФЗ. Полное соответствие требует и юридических действий со стороны школы как оператора ПДн.

### 3.1 Регистрация школы оператором ПДн в Роскомнадзоре

Обязательное действие. Школа собирает ПДн (имя, телефон, email клиентов) — значит является **оператором персональных данных** в смысле 152-ФЗ. Нужно подать **уведомление в реестр операторов ПДн** через [Госуслуги](https://www.gosuslugi.ru/) или сайт РКН. Подача бесплатная, рассмотрение 30 дней.

Без этого статуса любая обработка ПДн — нарушение, штрафы для юрлица сейчас от 100 тыс. до 18 млн ₽ за повторное.

### 3.2 Согласие на обработку ПДн от каждого клиента

В первом сообщении бот даёт краткий PII-disclaimer («передам контакт менеджеру»). Это **базовое** согласие. Для прода желательно усилить:
- разместить **полный текст** «Согласие на обработку ПДн» на сайте школы
- в боте давать ссылку на этот текст
- хранить факт прочтения/согласия (опционально, для аудита)

### 3.3 Политика обработки ПДн на сайте

Обязательный документ для оператора — публикуется на сайте школы. Шаблон есть на [рк-н.рф](https://rkn.gov.ru/), нужно адаптировать под фактический процесс.

### 3.4 Ограничение использования OpenAI

Это **частичное** соответствие, а не полное. Даже после маскирования (п. 2.7) у нас остаются нюансы:

- **Не вся ПДн поддаётся регулярному маскированию.** Мы хорошо ловим телефоны/email/сохранённые имена. Но если клиент напишет «я работаю в ООО „Ромашка“ на улице Ленина 5» — это тоже может быть идентифицирующая информация, и regex её не зацепит. Юридическая позиция тут спорная: РКН пока не давал однозначной позиции по «обезличенным диалогам с LLM», и при проверке могут трактовать строго.
- **Сама маршрутизация трафика через США** (даже маскированных данных) — это технически международный трафик. С точки зрения буквы закона это не «обработка ПДн», но в случае разбирательства будет неудобно доказывать.
- **OpenAI как процессор** не подписывает DPA по российскому образцу. По европейскому GDPR — да, по 152-ФЗ — нет.

**Что это означает практически:** при базовой проверке РКН (по жалобе или плановой) у нас есть хорошие технические аргументы и архитектура. Но если регулятор займёт жёсткую позицию против любого использования OpenAI российскими операторами — нам придётся либо мигрировать на отечественную LLM (YandexGPT/GigaChat), либо получать письменное согласие от каждого клиента на трансграничную передачу.

Это **архитектурный риск**, который надо учитывать стратегически. Технически мы готовы переехать на YandexGPT в течение 3-5 рабочих дней, если такое решение будет принято.

## 4. Что мы НЕ сделали (явные отказы по ТЗ)

В этой версии нет:
- автозаписи на консультацию через календарь — только сбор контакта в CRM
- голосового интерфейса
- мультиязычности (бот общается только на русском)
- сегментации лидов глубже «горячий / не горячий»
- A/B-тестов формулировок
- админ-панели в вебе — всё управление через xlsx и `.env`
- ВКонтакте-бота
- интеграций с другими CRM кроме Bitrix24

Эти ограничения зафиксированы в исходном ТЗ и не являются недоделкой.

## 5. Эксплуатационная готовность

- Развёрнут на VPS, работает 24/7
- Автоматический рестарт контейнера при сбое (`docker compose`, `restart: unless-stopped`)
- Логи с ротацией (10 МБ × 5 файлов)
- Изоляция секретов в `.env` (вне git)
- CI/CD через GitHub Actions
- Гарантийный период — 1 неделя после официальной сдачи, в течение которого исполнитель устраняет обнаруженные баги
- 89 юнит-тестов закрывают логику (рабочие часы, праздники, нормализация телефонов, маскирование ПДн, парсинг xlsx, payload Битрикса, guardrails, CLI сброса памяти)

Дальнейшие правки контента (цены, преподаватели, расписание, тексты) **заказчик делает самостоятельно** через xlsx — без обращения к разработчику.

## 6. Рекомендации на ближайшее будущее

1. **В первую очередь:** подать уведомление в Роскомнадзор о статусе оператора ПДн (п. 3.1) и опубликовать политику обработки ПДн на сайте (п. 3.3) — это критичные юридические шаги.
2. Прогнать **полный чек-лист ручного тестирования** ([tests/manual_test_cases.md](tests/manual_test_cases.md)) на боевом боте — записать pass/fail по каждому пункту, передать список найденных недочётов в гарантийный период.
3. Через месяц работы — посмотреть статистику в Битриксе: сколько лидов передано, какая доля стала клиентами. По результатам — обсудить точечные доработки (например, более тонкую квалификацию по нишам).
4. Параллельно — провести **внутреннюю аудиторию для менеджеров** по работе со сделками от бота: какой формат резюме им удобен, как реагировать на «звонок от лида с тегом из Telegram», какие частые вопросы клиентов всплывают (это сигнал для обновления базы знаний).
